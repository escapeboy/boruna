## Section 2 тАФ NEW Findings (missed or under-weighted by the prior pass)

**NEW-1 [HIGH, subsumed by 4.1 but worth calling out] тАФ Encryption can be *stripped* at verify (downgrade-to-plaintext), so "encrypted тЗТ tamper-evident" is defeatable.**
`verify.rs:132` gives encryption checks *only* when `manifest.encryption.is_some()`; there is no caller-supplied "this bundle must be encrypted" assertion and no external anchor binding the manifest. Scenario: attacker with write access to an encrypted bundle removes the `encryption` object from `manifest.json`, keeps the ciphertext files verbatim, and overwrites `file_checksums[f] = SHA256(ciphertext_bytes)`. `verify_bundle` (no KEK needed) then treats the files as plaintext, hashes match, `valid:true`. Net: the GCM per-file integrity is real *within* an encrypted read, but an attacker simply exits the encrypted read. Root cause is identical to 4.1 (manifest is attacker-rewritable, un-anchored). Fix requires the same external anchor / signature, plus optionally an operator flag "require encryption with kek_id=X".

**NEW-2 [LOW] тАФ `rotate_bundle` manifest rewrite is not crash-durable (weaker than `finalize`).**
`rotate.rs:153-156` writes `manifest.tmp.{pid}.json` with plain `std::fs::write` then `std::fs::rename`, with **no `fsync` of the temp file and no parent-dir fsync** тАФ unlike `finalize`, which uses `F_FULLFSYNC` + parent-dir fsync for `bundle.json` (`evidence.rs:326-356`). `rename` is atomic within a filesystem (no torn manifest), but on power loss the new manifest bytes may not have reached stable media, potentially reverting to the pre-rotation wrapping or an empty file. Availability/durability of the compliance record, not an integrity break. The module doc claims "no half-rotated states" (`rotate.rs:16-18`) тАФ true for atomicity, silent on durability.

**NEW-3 [LOW] тАФ `add_step_output` composes bundle-relative paths from `step_id`/`name` with no traversal guard.**
`evidence.rs:132-137` builds `outputs/{step_id}/{name}.json` and `bundle_dir.join(filename)` then `create_dir_all`+`write`. If `step_id`/`name` contained `..`/separators, writes could escape `bundle_dir`. Operator/workflow-authored (step ids come from the validated workflow def, not attacker input), so build-side and low, but it is the one place in the audit slice that joins un-sanitized name components into a filesystem path тАФ mirrors the `ref_to_run_id` gap (6.1) and the project's documented PatchBundle/ContextStore `..` hardening that this module does not apply.

**NEW-4 [LOW, confirms 3.6] тАФ plaintext manifest is a confirmation oracle even for encrypted bundles.**
`manifest.json` (incl. `file_checksums = SHA256(plaintext)`) is always written unencrypted (`evidence.rs:234-236`), regardless of the encryption block. For low-entropy / guessable step outputs an auditor-or-attacker can confirm content by matching `SHA256(guess)` against the checksum. Confidentiality only; noted here because it survives encryption.

**NEW-5 [INFO] тАФ `hostname`/`rust_version` in the compliance fingerprint are env-spoofable / usually "unknown".**
`fingerprint.rs:26-37`: `rust_version` = `option_env!("RUSTC_VERSION")` (no build script observed sets it тЖТ `"unknown"`); `hostname` = `$HOSTNAME`/`$HOST` or `"unknown"`. Both are part of the evidence record and trivially forgeable/absent. Low, but weakens the "environment attestation" value of the bundle.

### Bottom line

All four headline claims (F1/4.1, F2, F9/3.4, 3.2-4.2) are **CONFIRMED-ACCURATE**: for plaintext bundles `verify_bundle` is purely self-referential over an attacker-rewritable manifest, with no keyed hash and no external anchor тАФ forgeable. The two "SAFE" crypto claims survive scrutiny (`unwrap` panic-free = accurate; GCM per-file tag genuinely checked) **except** that the "encrypted тЗТ tamper-evident" framing is **understated**: NEW-1 shows an attacker can strip the encryption block and re-pass verification as plaintext, because verify has no encryption-required enforcement тАФ the same missing-anchor root cause as 4.1. The `..` traversal (6.1) is confirmed but bounded to a single level and latent (no CLI `get` sink). Nonce-reuse (5.2) is a real unenforced invariant whose at-rest exploitability is narrower than stated (overwrite keeps only the last ciphertext).
