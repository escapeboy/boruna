## What worked

1. **Splitting 0.5-S2 into a/b/c/d/e was the right call.**
   This sprint is genuinely small (~660 LOC including docs +
   tests) because the previous 4 sprints did all the
   protocol + infrastructure work. The submit-only path is
   ~100 lines of plumbing on top of two hardened halves.

2. **The `prepare_persistent_run` / `execute_after_insert`
   split paid back.** The existing `run_persistent` was
   already factored into "set up the run" + "execute the
   wave loop" methods. Adding submit-only was just an
   `if options.submit_only { return early; }` branch
   between them. No `run_persistent` refactor needed.

3. **Embedding `step_sources` in metadata avoids the
   coordinator/worker filesystem-access problem.** Workers
   on remote hosts don't need access to `workflow_dir`;
   they get the source via `metadata_json.step_sources`
   served by the coordinator. The 0.5-S2b coordinator
   already read this field; this sprint just formalizes
   the schema.

4. **Adversarial review caught the F1 silent-routing bug.**
   `--submit-only --skip-if-running` would have routed
   through the in-process path and silently violated
   operator intent. Reviewer flagged within seconds; clap's
   `conflicts_with_all` is a one-line fix.

5. **The marquee end-to-end integration test is genuinely
   end-to-end.** Spawns a real coordinator + worker, runs
   `boruna workflow run --submit-only` against a real
   `workflow.json` + `.ax` file, and asserts the step
   transitions Pending → Running → Completed via the
   worker with output_json matching expected. If the wire
   protocol or any persistence path breaks, this test fires.

## What didn't work / surprises

1. **The bulk-edit script collided with one pre-existing
   manual edit.** I'd manually added `submit_only: false`
   to `synthesized_options` BEFORE running the script,
   which then added another. Caught by `cargo build` (E0062
   "field used more than once"). 30-second fix but
   surfaces a class issue: when bulk-edit-meets-manual-edit,
   the script's "find pattern, append after" doesn't notice
   if the field is already there.

2. **Approval-gate `StepKind` field names changed from what
   I expected.** I wrote the test fixture with
   `approvers: vec![...]` and `description: "..."` but the
   actual variant has `required_role: String` and
   `condition: Option<...>`. Two compile errors, two
   one-line fixes.

3. **The `WorkflowStarted` audit event needs explicit
   emission across the early-return path.** I would have
   left it out (it lives inside `execute_after_insert`)
   except the design doc reminded me. Future audit
   reviewers would have flagged a chain with no genesis.
   **Pattern: when an early-return bypasses normal flow,
   audit emissions must be mirrored explicitly.**

## Decisions worth remembering

1. **`WorkflowStatus::Running` is the submit-only return
   status.** Reusing instead of adding a `Submitted`
   variant. Operators see "Running" with all steps
   "Pending" — semantically "in flight, not yet started by
   workers." Acceptable today; could split if confusion
   emerges.

2. **`metadata_json.step_sources` is OPERATIONAL ONLY** per
   convention #15. `workflow_hash` already covers source
   content for replay/audit. Embedding bodies again in
   metadata is a transport convenience, not a
   replay-verified record. Documented inline.

3. **Submit-only rejects approval-gate / external-trigger
   in the first wave.** Future sprint can lift this when
   distributed mode has a story for those (likely the
   workflow runner itself, never the coordinator/workers,
   handles them — they're operator-action-driven).

4. **256 KiB per-step + 4 MiB aggregate caps on
   step_sources.** Big enough to cover comfortably-large
   `.ax` sources (most are <10 KiB) and ~16-step workflows
   with large bodies. Small enough to keep
   `metadata_json` round-trippable through SQLite and HTTP.
   Documented inline.

5. **`--submit-only --concurrency >1` is silently ignored**
   (in-process workers aren't spawned). Acceptable for
   MVP; future warn-at-submit would close the gap. Not a
   correctness bug, just a UX papercut.

## Recommendation for next session

**Strong: stop here for the day.** Six sprints in one session
is enough; the 0.5.0 cycle's marquee distributed-execution
stack is now end-to-end functional for single-wave workflows.

**If continuing: 0.5-S2f — multi-wave advancement.** Pick
one of:
1. **Client-side `boruna coordinator wait <run-id>`** — a
   read-only client that polls the coordinator's
   `/api/runs/:id`, reads workflow_def from metadata,
   computes downstream-ready successors after each
   completion, writes more Pending checkpoints. Keeps the
   coordinator dumb per the 0.5-S2c architecture note.
2. **Coordinator-side advancement** — when a step
   completes via `/api/work/complete`, the coordinator
   reads `metadata_json.workflow_def` (would need to embed
   the def in metadata) and marks downstream-ready steps
   as Pending. Eliminates the "client must stay running"
   constraint but changes the coordinator's role.

Both are small-medium sprints. Option 1 is more in keeping
with the architecture established in 0.5-S2c.

## Patterns added this sprint

- **CLI flag mutual-exclusion via clap `conflicts_with_all`.**
  When flags are semantically incompatible, declare at
  parse time. Less surface for silent-routing bugs.
- **Per-item + aggregate size caps for embedded payloads.**
  Cap both dimensions; one-cap allows pathological skew.
- **Audit-event mirroring across early-return paths.** When
  a new code path bypasses normal flow, mirror the audit
  emissions explicitly so the chain has its genesis.
- **`prepare-X` + `execute-Y` factoring as the early-return
  hook.** When a function naturally splits into "set up
  state" + "execute work," refactor to two helpers BEFORE
  adding new modes. Future modes (submit-only,
  resume-only, dry-run, ...) become one-line branches
  between them.
