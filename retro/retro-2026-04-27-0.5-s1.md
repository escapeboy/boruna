## What worked

1. **Convention #27 paid back.** The ADR-only sprint pattern from
   ADR 001 (sprint 0.3-S1) translated cleanly to this sprint. No
   separate Think doc, no separate Plan doc, no Build phase. One
   document captures everything; the reviewer focuses on
   decision quality, not document structure.

2. **Adversarial review on an ADR is at least as valuable as on
   code.** `ce-adversarial-document-reviewer` raised 6 HIGH
   findings, every one of which was load-bearing for the
   architectural shape:
   - Lease re-dispatch race window (added `claim_id` + CAS)
   - Coordinator restart undefined (committed to "all leases
     void")
   - Cross-step capability budgets silent assumption
   - Atomic-upgrade rule was buried in non-binding notes
   - `--concurrency` semantics was a decision masquerading as
     guidance
   - Output payload size unspecified (locked 8 MB)

   The ADR is materially stronger after this pass. **Pattern
   reinforced: adversarial review is a per-sprint gate, not just
   a per-code-sprint gate.**

3. **The constraints from ADR 001 still bind.** Single-binary
   distribution, embeddable in other processes, local-first,
   determinism. Reusing them as inherited constraints kept the
   alternatives section tight — most alternatives were rejected
   on a single inherited constraint, not a new criterion.

4. **Reusing existing infrastructure decisions was free design
   leverage.** The `serve` feature flag, the dashboard's HTTP
   scaffolding, the `protocol_version` convention, the
   policy-validate single-parser pattern, the `capability_set_hash`
   from 0.3-S15 — all flowed into the ADR without needing
   re-derivation. **Convention reinforced: prior architectural
   choices compound when they're documented and discoverable.**

5. **Implementation notes section was the right escape valve.**
   Tunable parameters (long-poll window, lease TTL) belong as
   non-binding guidance for the next sprint. The reviewer's
   convention-#28 enforcement (no implementation snuck into the
   ADR) was the right tension — it forced me to distinguish
   *binding decisions* from *recommendations*.

## What didn't work / surprises

1. **The first draft buried two real decisions in non-binding
   guidance.** Note 9 (`--concurrency N` semantics) and the
   atomic-upgrade rule were both effectively decisions but
   labeled as suggestions. The adversarial reviewer caught both.
   **Lesson: when writing implementation notes, ask "would
   changing this in the next sprint break a contract?" If yes,
   it's a decision, not a note.**

2. **Capability budgets across steps was an unstated
   assumption.** I treated policies as per-step stateless without
   saying so. Distribution silently breaks any future cross-step
   budget. The fix was a single explicit constraint —
   "per-step-stateless policy enforcement" — but it took an
   external reviewer to notice the gap. **Lesson: when a design
   commits to distribution, every property that depends on
   single-process state must become an explicit constraint or
   an explicit non-goal.**

3. **No alternatives outside the BYOH frame considered.** The
   reviewer flagged Kafka, WASM-in-process, and async tasks as
   missing alternatives. I judged none rise to high-confidence
   omissions (they all fail one of the inherited constraints),
   but the reviewer's framing was right: I should at least name
   them, even to dismiss them. The current ADR's Considered
   Alternatives section is tight; an "explicitly considered and
   rejected" subsection might be worth adding to a future ADR.

## Decisions worth remembering

1. **`claim_id` + CAS is the cheapest way to close the
   slow-but-not-dead worker race.** Without it, the
   `status == 'Running'` check has a ~millisecond hole between
   re-enqueue and re-claim where a late `complete` POST could
   succeed. The state-machine fix is one extra column on
   `step_checkpoints` and one extra `WHERE` clause in the
   handler.

2. **Treat coordinator restart as a partition.** Persisting
   in-memory lease state across restarts buys little (re-claim
   costs are milliseconds; coord crash is rare-ish) and adds a
   second consistency surface. Simpler model wins for v0.5.0.

3. **Atomic-upgrade requirement is the deliberate v0.5.0
   posture.** Heterogeneous binary versions add per-capability
   version negotiation that is its own design. Operators
   schedule a maintenance window for upgrades; rolling upgrades
   are 0.6.x+. Stating this loudly avoids surprising operators
   at adoption time.

4. **8 MB output cap is honest.** Without a stated cap, the
   protocol commits the coordinator to buffering arbitrarily
   large bodies forever. 8 MB is the max-LLM-output-shape +
   structured-data-shape envelope; a future content-addressed
   blob backend handles bigger payloads via the reserved
   `output_blob_ref` field.

## Recommendation for next session

**0.5-S2 — Implementation of ADR 002.** Per the ADR's own
implementation notes:

- New modules `crates/llmvm-cli/src/coordinator.rs` and
  `worker.rs` next to `dashboard.rs`. Both behind the `serve`
  feature flag.
- Schema migration: add `worker_id`, `lease_expires_at`,
  `claim_id`, `claim_count` columns to `step_checkpoints` via
  the v1→vN runner. Old binaries reading new schemas see
  `#[serde(default)]` ignored extras.
- Wire format: JSON, every response carries
  `protocol_version: 1` (convention #4).
- Stable error_kind taxonomy: `coord.lease_expired`,
  `coord.unknown_worker`, `coord.workflow_completed`,
  `coord.binary_mismatch`, `coord.invalid_request`,
  `coord.output_too_large`. Locked at first ship.
- Integration test: spawn coordinator + 1-3 workers in a
  tempdir, run small DAG end-to-end, kill a worker mid-step,
  assert lease expiry + re-dispatch + bit-identical
  `output_hash`.

Estimated size: **medium** (one full sprint). Risk surface is
the state machine — adversarial review on the implementation
will validate whether the `claim_id` CAS actually holds under
race conditions.

## Patterns added this sprint

- **For protocol-defining ADRs, lock the route surface first
  and the field schemas later.** This ADR commits to the
  six route paths and the lease state machine; the JSON
  field-level shapes are deferred to the implementation
  sprint. The route-level commitment is enough to lock the
  shape; field-level commitment is something the implementer
  can iterate on with adversarial review.

- **Inherited constraints from prior ADRs become decision-
  criteria multipliers.** ADR 002's rejected-alternatives
  section is tight because most alternatives lose on a single
  ADR-001-inherited constraint (single binary, no daemon,
  determinism). Future ADRs should reference inherited
  constraints by name.

- **Output payload caps in protocol ADRs are mandatory.**
  Without a stated max body size, a route protocol silently
  commits to "arbitrary." Lock at first ship.
