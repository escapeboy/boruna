## What worked

1. **Splitting 0.5-S2 into a/b/c/d kept each sprint genuinely
   small.** Persistence (a) → HTTP MVP (b) → background sweep
   (c) → listener-merge (d). Each ships in roughly one session.
   The discipline of not letting scope creep into "let me also
   add X" has been load-bearing for the whole 0.5.0 cycle.

2. **Public route-builder pattern.** The refactor exposing
   `dashboard::dashboard_routes(store, bind_warning)` as a
   `pub` fn taking primitive args (Arc, Option<String>) instead
   of the internal `DashboardState` was the cleanest design
   choice. The coordinator gets exactly the contract surface
   it needs; the dashboard's internal state shape stays
   private and free to evolve. **Add to convention list:**
   when a Router needs to be embedded via `merge`, expose a
   pub builder taking primitive args, not the internal state
   struct.

3. **The "both routes on same listener" regression test was
   load-bearing.** Adversarial review pointed out that the
   first 3 tests each exercised dashboard OR coord routes
   against a coord, never both against the SAME running
   instance. The new
   `coord_serve_handles_both_coord_and_dashboard_routes_on_same_listener`
   test interleaves register → api-runs → heartbeat → index
   calls against one port. If the merge ever silently breaks
   one of the route trees, this test catches it.

4. **Adversarial review confirmed the merge correctness.** Per-
   router layer scoping (DefaultBodyLimit applies to coord
   routes only), single store handle (Arc-cloned, both routers
   see same lock), and bind_warning propagation all reviewed
   clean. Three LOW residual risks raised, all documented or
   addressed.

## What didn't work / surprises

1. **Adding 4 tests to the integration file caused parallel-
   load flakiness.** The `coord_bg_sweep_requeues_expired_lease`
   test with 200 ms sweep interval started flaking under the
   12-test parallel load. Fixed by replacing the fixed 600ms
   sleep with poll-with-deadline (5s). The
   `worker_subprocess_completes_step_end_to_end` test also
   flaked once under load but passed on retry — accepted as
   transient.

   **Lesson:** integration tests that spawn child processes
   compound CPU pressure quickly under cargo's parallel test
   runner. Either (a) make every test poll-with-deadline
   (more robust); (b) annotate `#[serial_test]` (slower but
   deterministic); (c) reduce `--test-threads` for this file.
   Accepted (a) for now; (b)/(c) deferred.

2. **The `}` cleanup after editing the route builder.** I
   added a comment block to the `coord_router` definition,
   then accidentally left a stray `}` from the old structure.
   The compiler caught it immediately. Standard issue when
   editing nested blocks; in retrospect, reading the
   immediate context before the edit would have caught it
   pre-write.

3. **Clippy's `result_large_err` and `dead_code` lint
   interactions.** Removing `#[allow(dead_code)]` from the
   `bind_warning` field (now used in the merge) was easy. The
   `clippy::result_large_err` lint stays from 0.5-S2b on
   `try_claim_one` and `validate_session`. No churn this
   sprint.

## Decisions worth remembering

1. **DefaultBodyLimit applies to coord routes only.** Per
   Axum's per-router layer scoping. Documented inline. Future
   POST dashboard routes must opt in explicitly OR be added
   to coord_router instead.

2. **Both routers share `Arc<Mutex<RunCheckpointStore>>`.**
   Single connection, single mutex. Adversarial review
   flagged this as RR-1 (shared serialization could starve
   workers under heavy dashboard load) but it's acceptable
   at v0.5.0 scale. Future tuning: connection pool or
   read-replica.

3. **The merge is automatic.** Operators running
   `boruna coordinator serve` get the dashboard routes too.
   No flag, no opt-in. Standalone `boruna dashboard serve`
   keeps working for read-only deployments.

4. **Standalone path stays.** Refactoring the dashboard's
   `run_serve` to call the new builder internally means the
   existing `cli_dashboard.rs` integration tests didn't need
   changes — proof that the standalone path works
   identically.

## Patterns added this sprint

- **Public route-builder pattern for sub-routers.** When a
  Router will be embedded via `merge`, expose a `pub` builder
  fn taking primitive args (Arc handles, Option<String>
  config) rather than the internal state struct. Decouples
  the consumer from the internal state shape; makes the
  contract minimal.
- **Per-router layer scoping documentation.** Layers attached
  pre-merge stay bound to their own routes in Axum. When
  this matters (e.g., body limits), document inline so a
  future route addition doesn't accidentally inherit/lose
  a layer.
- **Multi-route-tree regression test.** When merging routers,
  add at least one test that exercises BOTH route trees
  against the same running instance. Separate-test coverage
  isn't enough; explicit interleaving proves the merge.
- **Poll-with-deadline over fixed-sleep for timing-sensitive
  integration tests.** Especially when adding more
  parallel-test load to an existing file. Fixed sleeps were
  fine for 6-test files; with 12 tests they flake.
