## Goal

The 0.5-S7 retro recommended a small followup sprint to switch the
dashboard's per-run detail page to use the new
`read_step_output` accessor, on the framing that "the dashboard
reads `cp.output_json` directly." Audit during the Think phase
revealed the framing was wrong: the dashboard HTML never rendered
output bodies at all (since 0.4-S16). The actual gap was a UX one
— operators couldn't see step outputs from the dashboard without
shelling out — and the blob-aware piece is "for offloaded outputs
emit a link to the blob route, don't slurp bytes."

Re-scoped to: add an `Output` column to the per-run detail HTML
table that renders inline-truncated text or a blob link as
appropriate.

## What shipped

- `dashboard.rs::render_detail` gains an `Output` column.
- New `StepOutputDisplay` enum (`None | Inline(String) | Blob(String)`)
  decouples the lookup from the rendering.
- New `resolve_step_outputs_for_render` resolves per-step outputs
  through `read_step_output` for inline cases; for blob-stored
  steps it short-circuits to the ref so we never pull multi-MB
  bytes into a dashboard render.
- New `render_output_cell` builds the cell:
  - Inline: `<code>{truncated, html-escaped}</code>` + `…` if truncated
  - Blob: `<a href="/api/runs/{run_id}/blobs/{hash}"><code>[blob: hash[..16]…]</code></a>`
  - None: `—`
- Defensive: malformed `output_blob_ref` (not 64 lowercase hex) falls
  back to plain `<code>[blob: ...]</code>` text — no broken link.
- New `truncate_chars` walks `char_indices` for UTF-8 boundary safety.
- 5 new dashboard unit tests:
  - inline render with html escape
  - blob link render with full hash in URL + short label
  - em-dash for pending steps
  - `truncate_chars` boundary cases (0, exact, multi-byte)
  - malformed blob ref fallback

The JSON API endpoint is unchanged: `StepCheckpoint` already
serializes both `output_json` and `output_blob_ref` since 0.5-S7,
so programmatic consumers can branch on the shape with no server
change.

## What worked

### 1. The Think-phase audit caught a wrong premise

The S7 retro had said "the dashboard reads `cp.output_json`
directly." Spent 20 minutes auditing the actual code before
writing any new lines. Discovered the dashboard NEVER rendered
outputs in HTML; the JSON endpoint already exposed both columns
correctly. Rewrote the design doc with the corrected framing
("UX gap, not silent-failure bug") and the scope shrank from
"swap accessor everywhere" to "add an Output column with
blob-aware rendering."

**Lesson:** retro recommendations are write-time best guesses.
When the next session picks them up, the Think phase should
include an explicit "verify the premise" step. The retro was
written 30 minutes after S7 shipped; my mental model was off.

### 2. Blob-link path skips byte fetch

Crucial design decision: when a step is blob-stored, the renderer
gets the **ref**, not the bytes. We only call `read_step_output`
for the inline / no-output cases. A page rendering 50 completed
steps where each output is 5 MB would otherwise pull 250 MB into
the dashboard render. With this design, the dashboard cost is
unchanged regardless of blob size.

Encoded in `StepOutputDisplay::Blob(hash)` carrying just the hash;
the renderer emits a link without ever invoking the BlobStore.

### 3. Adversarial review came back clean

`ce-correctness-reviewer` had **zero HIGH/MEDIUM findings** —
which on a UI render is meaningful, because HTML injection is
the obvious risk surface. Confirmed:
- HTML escape applies to run_id (in href attribute) and hash.
- Hex validation gates the blob path before URL generation.
- `truncate_chars` walks char boundaries; verified at 0, exact,
  multi-byte boundary cases.
- Lock discipline matches the existing `load_run_detail` pattern.
- Disclosure level matches the JSON endpoint.

Two LOW residual risks (mutex contention under load, run_id URL
encoding) are pre-existing patterns from 0.4-S16 — not introduced
by this sprint and not in scope.

Two testing gaps the reviewer noted (truncate boundary cases,
malformed blob fallback) were cheap to close — each is a 5-line
test. Closed both.

### 4. Two clippy churns from the toolchain skew

Local rustc 1.91.1 emitted two new lints on my new code:
`explicit_counter_loop` (manual `count` variable) and
`needless_borrow` (`&truncated` where `truncated` already
implements the trait). Fixed both in 30 seconds. **Pattern:**
when adding new code, run `cargo clippy` BEFORE writing tests so
new lints surface during build, not during the gate phase.

## What I'd do differently

### 1. Spent 20 minutes on a misframed scope

If I'd run the audit at S7-retro-write time, the retro would have
recommended "S7b: add an Output column" rather than "S7b: swap
accessor in dashboard." The corrective audit cost ~20 minutes;
not catastrophic, but a good prompt to add a "verify retro
premise" step to the Think phase template.

### 2. Should propose smaller iteration units

This was a 50-line code change but I went through the full
sprint pipeline (Think + Plan + Build + Review + Test + Ship +
Reflect). The pipeline overhead probably equals the actual
implementation time on sprints this size. For micro-sprints
(<100 lines), `quick` mode (Build → Review → Test → Ship) is
the right shape. Used `full` only because the user invoked
`/sprint-orchestrate full` — but the orchestrator could
automatically demote to `quick` when the plan-phase output is
shorter than X lines or covers a single file.

## New patterns

### §40 — When picking up a retro recommendation, audit the premise first

Retro recommendations are write-time guesses about what's broken.
The next sprint's Think phase should explicitly verify the
recommendation against the current code before committing scope.
The cost is ~20 minutes; the benefit is avoiding building the
wrong thing.

**Demonstrated:** S7's retro said the dashboard read
`cp.output_json` directly. The actual code didn't render outputs
at all. The corrective audit reframed the sprint from a
correctness fix to a UX add.

### §41 — When rendering remote-fetchable resources, render the ref, not the bytes

When a UI page references content that lives behind a separate
fetch route (blob store, CDN, S3), the page should emit a link
to the resource — not slurp the bytes into the page render. This
keeps the page's render cost independent of the resource's size.

**Demonstrated:** dashboard's `StepOutputDisplay::Blob(hash)`
carries the ref; the renderer emits an HTML link to the S7 blob
route without ever touching the bytes.
