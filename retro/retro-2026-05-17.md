# Sprint Retro — agent-native-cli (2026-05-17)

**Sprint:** 5 agent-DX surfaces borrowed from a `vercel-labs/zero` competitive review.
**Branch:** `feature/agent-native-cli` → PR #68 (base `master`).
**Outcome:** Shipped. 19 files, +1749 lines. All gates green except one verified pre-existing failure.

## What shipped

`boruna lang codes`, `boruna doctor`, `boruna workflow graph`, `boruna size`,
`boruna skills` — all read-only, all `--json`. Diagnostic-code registry with a
source-parsing drift test. 18 new tests (12 integration + 6 unit).

## What went well

- **Premise audit before scoping (convention §40).** The research's idea #1
  ("add stable diagnostic codes") was already implemented — `E001`–`E009`
  existed. Auditing the code first turned it into a *registry* feature (the
  genuinely-missing agent-facing piece) instead of dead work.
- **Sequential was correct (convention §31).** All 5 surfaces edit
  `crates/llmvm-cli/src/main.rs` (shared enum + dispatch). Sequential build,
  no parallel agents, zero merge friction.
- **Verified, didn't assume, on the failing test.** `cli_watch` failed; rather
  than hand-wave it as "flaky", stashed the branch and reproduced it on clean
  `master` — confirmed pre-existing.

## What went wrong

- **`git pull` on the wrong branch.** `master` was checked out in an agent
  worktree, so `git checkout master` failed silently; the subsequent
  `git pull origin master` ran on `future/model-eval` and started an unwanted
  merge with conflicts. Recovered with `git reset --hard origin/master` on the
  fresh branch (zero real work lost). **Fix for next time:** when `master` is
  held by a worktree, branch directly — `git fetch origin master` then
  `git checkout -b <branch> origin/master`. Never `pull` onto a feature branch.
