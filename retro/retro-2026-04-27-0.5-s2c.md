## What worked

1. **Splitting 0.5-S2 into a/b/c/(d) keeps each sprint focused.**
   Persistence (a) → HTTP (b) → background sweep (c) → runner
   integration (d). Each ships in roughly one session with a
   focused test target. The "split-and-conquer" cadence has
   been the dominant pattern for the 0.5.0 cycle and is paying
   back as predicted in the 0.5-S1 ADR retro.

2. **Architectural note "coordinator is a dumb transport" set
   expectations clearly.** The design doc explicitly states
   that wave advancement is the client's job in v0.5.x. This
   pre-empts the obvious question "why doesn't the coordinator
   know about workflow DAGs?" — it's deferred to 0.5-S2d
   intentionally.

3. **The 2-step DAG test is small but proves the protocol.**
   Pre-populating both steps as Pending up front isn't the
   real workflow shape, but it proves a single worker can
   handle multiple sequential claims without protocol
   weirdness. The test is deterministic and runs in <3s.

4. **Adversarial review caught 2 actionable cosmetic-but-real
   bugs.** F1 (sweep threshold drift) is a 1ms boundary
   inconsistency that would produce flaky tests if anyone ever
   relied on the boundary semantics. F2 (poisoned mutex
   silent skip) is a diagnosability gap — if a handler ever
   panics while holding the store mutex, the sweep silently
   stops working and operators have no signal until they
   notice the original panic. Both fixes are tiny.

5. **`tick.tick().await` first-skip pattern is genuinely
   important.** Without it the FIRST tick fires immediately
   (at t=0), which would re-do the work the startup sweep
   just did 100ms ago. Documented in the in-method comment.

## What didn't work / surprises

1. **`tokio::time::interval` defaults are non-obvious.** The
   default `MissedTickBehavior::Burst` means a long-running
   sweep that misses ticks fires multiple times in rapid
   succession to "catch up." Acceptable for our use case
   (sweep is idempotent + fast), but worth knowing for future
   work. Documented as a residual risk in the review.

2. **The `Ok(0)` quiet logging is correct but interacts
   badly with `Err`-loud logging under sustained errors.** F3
   from review: under a persistent DB failure, the sweep prints
   ~5 lines/sec to stderr indefinitely with `--sweep-interval-ms=200`.
   Acceptable for v0.5.0 (operators must notice failure) but
   flagged for future hardening. A simple rate-limit or
   exponential-backoff would close it.

3. **Tokio task abort safety required reading the docs
   carefully.** The reviewer's confirmation (info-level) was
   load-bearing: abort fires at `.await` points only, not
   mid-block. Combined with the std::sync::Mutex serialization
   and the SQLite `BEGIN IMMEDIATE` / `COMMIT` boundary, this
   means the sweep can't be torn mid-transaction. But this
   chain of reasoning isn't obvious; documenting it inline
   helps future readers.

## Decisions worth remembering

1. **`MIN_SWEEP_INTERVAL_MS = 100` with auto-clamp + warning.**
   Rejecting at clap parse-time would require a custom
   value_parser. Clamp + warning is the simpler choice and
   matches the project's "reject at operation time" convention
   (#1) — operators see the warning at startup and know they
   asked for something below the floor.

2. **Background sweep uses `now_ms + 1` matching startup.**
   F1 fix. Both paths now treat `lease_expires_at == now_ms`
   as expired. Documented in the in-loop comment.

3. **Poisoned-mutex log-once.** Continuous logging would
   turn one panic into log-spam noise; silent skipping
   forever would mask the failure entirely. Log-once on
   first poison transition gives operators ONE signal in
   stderr that they can correlate with the original panic.

4. **Coordinator is a "dumb transport" in v0.5.x.** This is
   the architectural choice that makes the sprint sequence
   tractable. 0.5-S2d will revisit when adding the wave-loop
   integration.

## Recommendation for next session

**0.5-S2d — Workflow runner integration.** The protocol is
locked, the persistence is hardened, the HTTP plumbing works,
and the background sweep recovers from worker crashes. The
remaining piece is wiring the existing `WorkflowRunner` to
dispatch via the coordinator instead of `std::thread::spawn`.

Approach: extend `boruna workflow run` with a `--coordinator
<url>` flag. When set, the runner:
1. Validates the workflow + computes the DAG (existing
   path).
2. Inserts run row + initial wave's step Pending checkpoints
   into runs.db.
3. Polls runs.db for step terminal states; on each
   completion, computes downstream-ready successors and
   marks them Pending (the wave loop, but driven by polling
   instead of `JoinHandle`).
4. Exits when all steps reach a terminal state.

Estimated size: medium. The protocol design is done; this is
integration plumbing. Risk surface is partial-wave-failure
handling (per 0.4-S7's multi-pause logic) and the
client-coordinator polling cadence.

## Patterns added this sprint

- **Tokio task abort safety with std::sync::Mutex.** Acquire
  lock, do synchronous work, drop lock, THEN `.await`. abort
  cancels at `.await` only, never mid-lock.
- **Poison-mutex once-only logging.** Track logged-state
  in the task so a poison condition logs ONCE (diagnostic)
  instead of silently or noisily forever.
- **First-tick skip after `tokio::time::interval`.** The
  immediate-fire is consumed BEFORE the work loop, so the
  first work happens at `interval_ms` instead of `t=0`.
  Useful when there's a redundant startup pass.
