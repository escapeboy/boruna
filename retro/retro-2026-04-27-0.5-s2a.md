## What worked

1. **Splitting 0.5-S2 into a/b mirrored the 0.3-S2a/b precedent
   exactly.** ADR 001's split — schema + module + tests as one
   sprint, runner integration as the next — paid back well per
   the 0.3-S2 retros. This sprint repeated the pattern: focused
   API surface, no HTTP surface area, every race condition
   covered by an explicit test. The next sprint becomes mechanical
   wiring on top of a hardened base.

2. **The flagship `slow_worker_race_late_completion_rejected`
   test is the assertion the entire distributed-execution
   surface depends on.** Eight steps in one test, each step is a
   precise state-machine transition. If any of: lease expiry,
   re-claim claim_id increment, late-completion rejection, or
   row-state preservation breaks, this test fires. **Pattern
   reinforced:** when a sprint commits to a single load-bearing
   property, write one test that asserts that property
   end-to-end. The unit tests cover individual methods; the
   flagship test covers the property.

3. **Adversarial review caught 5 findings; 3 were correctness
   bugs that would have shipped silently.** Every one was
   load-bearing for the distributed-execution surface:
   - **F1** — CI's default-features build masked a feature-gated
     compile error. Found via `cargo check -p boruna-cli
     --features serve`.
   - **F2** — `unchecked_transaction()` issues `BEGIN DEFERRED`,
     not `BEGIN IMMEDIATE`. Architecture doc said the latter.
     Code didn't match. Refactored.
   - **F3** — `upsert_step_checkpoint` didn't clear claim
     columns. Mixed-mode hazard: stale `lease_expires_at` from
     a prior distributed-mode claim would let
     `expire_leases_and_requeue` re-trigger mid-execution
     against a live single-process attempt.
   - **F4** — single-connection tests don't exercise the actual
     CAS guarantee under contention. Added two multi-connection
     stress tests.
   - **F5** — `complete_step_cas` clobbered `error_msg` to NULL
     on success. Diagnostic-data loss path. Changed to COALESCE.

   **Pattern reinforced (#29):** adversarial review pays for
   itself. 24+ HIGHs across cycles, ~80% actionable. F3 alone
   would have been a real corruption bug shipping to production
   in 0.5-S2b.

4. **The python-script bulk edit got 17 of 19 sites right; the
   review caught the 2 it missed.** The script's logic — track
   brace depth, only insert at depth 1 inside `StepCheckpoint
   {` opens — handled the common case. The two misses were
   feature-gated code paths that didn't compile in the default
   build. **Lesson: any bulk edit by script needs the
   adversarial reviewer to look at the actual diff, not the
   "X edits" output, to catch what the script's pattern matcher
   missed.**

## What didn't work / surprises

1. **The git-checkout/script ping-pong cost me ~30 minutes.**
   First script run was over-aggressive (matched non-StepCheckpoint
   structs); revert; second script ran twice without revert
   between (resulting in duplicate field insertions). Should
   have used `git stash` instead of multiple revert+rerun cycles
   to keep the work tree consistent. Cleaner approach for
   future bulk edits: revert, run script ONCE, verify with
   `cargo check`, then proceed.

2. **`unchecked_transaction()` was a footgun.** I assumed it
   meant "transaction with whatever defaults" — actually it
   means "transaction that bypasses rusqlite's internal
   tracking." The BEGIN mode is still DEFERRED. The
   architecture doc and the code both said BEGIN IMMEDIATE.
   The reviewer caught the gap. **Lesson: when an architecture
   doc names a specific mechanism, the code should literally
   issue that mechanism — not rely on a wrapper's defaults.
   Verify by reading the wrapper's source or by running the
   code and observing the SQL.**

3. **`#[allow(clippy::too_many_arguments)]` on
   `complete_step_cas` was unavoidable.** The method has 8
   parameters because the caller-controlled fields all need
   to flow through (run_id, step_id, claim_id, output_json,
   output_hash, attempt_count, ended_at, lease_ttl). A struct
   would have been a wrapper around the same 8 fields with no
   real abstraction win. Took the lint waiver.

4. **The pre-existing 0.4-S14 metrics flake is still there.**
   Documented in carried-forward debts since 0.4-S14. Not
   blocking, but every sprint's full-workspace test run
   produces a noisy "1 failed" line that I have to manually
   ignore. Worth a small follow-up sprint to remove the
   `unsafe set_var` test.

## Decisions worth remembering

1. **`upsert_step_checkpoint` clears `worker_id` and
   `lease_expires_at` but preserves `claim_id`.** The lease is
   release-by-default semantics; the counter is monotonic.
   This means a step that flows through both modes (single-
   process upsert, then distributed claim) doesn't reset its
   claim_id to 0 — the distributed claim sees claim_id=N+1.

2. **`COALESCE(?8, error_msg)` on completion preserves
   diagnostic data.** A successful completion that follows a
   transient-error attempt no longer clobbers the error
   message. Useful for triage. The fail path explicitly passes
   `Some(error_msg)`, which COALESCE returns directly.

3. **`BEGIN IMMEDIATE` + manual COMMIT/ROLLBACK is the
   project-convention pattern, not `unchecked_transaction()`.**
   Documented in the in-method comment. Future state-machine
   methods should follow the same shape.

4. **Stable `kind()` strings are locked at this sprint's ship.**
   `claim.*`, `terminal.*`, `extend.*`. The HTTP layer in
   0.5-S2b maps these 1:1 to wire-level `coord.*` strings —
   neither side can drift independently.

## Recommendation for next session

**0.5-S2b — Coordinator HTTP + worker HTTP + integration test.**

The persistence API is the contract; the HTTP layer is the
wrapper. Per ADR 002's implementation notes:

- `crates/llmvm-cli/src/coordinator.rs` and `worker.rs`, behind
  `serve` feature flag (already wired for dashboard).
- Wire format: JSON, `protocol_version: 1` on every response
  (project convention #4).
- 6 routes: register, heartbeat, claim (long-poll), complete,
  fail, extend-lease. Plus dashboard's existing read routes
  coexist on the same listener.
- Stable `coord.*` error_kind taxonomy mapping the persistence
  outcomes 1:1.
- Integration test: tempdir + coordinator + 1-3 workers + small
  DAG + kill-a-worker-mid-step + assert lease expiry +
  re-dispatch + bit-identical `output_hash`.

Estimated size: medium. Risk surface is the long-poll dispatch
loop. Adversarial review on the implementation should
specifically check:
- Long-poll graceful-shutdown (no orphan tokio tasks).
- Worker crash mid-step → lease expiry → re-claim works
  end-to-end.
- 8 MB output cap is enforced with `413 Payload Too Large`.

## Patterns added this sprint

- **`claim_id` + atomic CAS** as the canonical lease state
  machine for distributed work claiming. Documented in
  schema_v1.sql comments + the design doc + the architecture
  doc + the in-method comment (project convention #18 — 4
  places).
- **Multi-connection stress tests for CAS guarantees.** Any
  future sprint that adds a CAS-protected method must include
  at least one multi-connection regression test (open N
  connections to a tempfile DB, race them, assert
  exactly-one-wins). The deterministic single-thread tests are
  necessary but not sufficient — they don't exercise the
  actual contention path.
- **`error_msg = COALESCE(?, error_msg)` for non-error
  transitions.** Preserves diagnostic data across attempts.
  Generalize to any column where mid-attempt data carries
  forward to terminal state.
