## What worked

1. **Empirical validation before committing the ADR was load-bearing.** Built a `/tmp/rusqlite-musl-probe/` project and confirmed `rusqlite/bundled` linked statically (otool -L showed only `libSystem.B.dylib`, no `libsqlite3`). Took 2 minutes; converted "I think this works" to "I built it and verified the link type." The ADR's musl-cross claim is now honestly framed as "host build proves bundled works in principle; cross-compile deferred to first 0.3.x release with documented fallback."
2. **`ce-feasibility-reviewer` paid for itself.** Found 1 CRITICAL + 3 HIGH + 3 MEDIUM real issues. Two of them were silent footguns: `PRAGMA foreign_keys = ON` is OFF by default in SQLite (so `ON DELETE CASCADE` would have been silently inert), and `started_at` columns conflict with the determinism contract unless explicitly bucketed as "operational" vs "replay-verified". Without the review, both would have shipped in the schema for `0.3-S2` and someone would have eaten the debugging cost a year later.
3. **The reviewer caught self-contradiction in my own ADR.** I labeled this a "decision sprint" but the Implementation Notes section made 7 concrete decisions that belonged in `0.3-S2`. The fix: explicitly frame those notes as "non-binding guidance for next sprint," and add a sizing flag recommending `0.3-S2` get re-sized from L to L+ or split. The ADR now does what an ADR should — pass the baton with enough context that the next sprint doesn't have to re-derive the gotchas, without pre-deciding the implementation.
4. **Naming the determinism contract explicitly is now a transferable artifact.** Future schema additions in `0.3-S2`–`0.3-S9` can reference this ADR's "Determinism contract for persisted state" section as the litmus test. The next reviewer doesn't have to re-litigate why `started_at` is fine but feeding it into a hash isn't.

## What didn't work / surprises

1. **I almost shipped a self-contradicting ADR.** Without the reviewer pass, the doc would have claimed "decision sprint" while embedding a fully-specified schema, PRAGMA list, and 7-step implementation plan. The reviewer's H2 finding ("ADR contradicts itself on what 0.3-S2 includes") forced the right framing. **Lesson: when writing decision-only artifacts, the reviewer's job is partly to enforce that you didn't sneak implementation in.**
2. **I missed that adding `rusqlite` to `orchestrator/Cargo.toml` would link SQLite into all 4 release binaries.** Reviewer's C1 finding caught this — `boruna-mcp` and `boruna-pkg` don't need persistence. Fix is a `[features]` gate so SQLite only lands in `boruna` (CLI) and `boruna-orch`. Saves ~3 MB across the release tarball. Easy to fix once flagged; would have been hard to roll back after `0.3-S2` shipped.
3. **The legacy `orchestrator::storage::Store` (JSON files) has its own determinism smell.** Discovered while reading `storage/mod.rs:64-89` for context: `latest_graph()` orders by filesystem `metadata.modified()` — wall-clock — to find the latest. Not addressed by this ADR (out of scope) but documented in the Open Questions section and flagged as something `0.4-S7` (dashboard backend) will need to deal with.
4. **`cross` not installed locally** — couldn't run the actual musl cross-compile validation in this sprint. Honest framing in the ADR: host build done, cross deferred to first release attempt, with a documented fallback (custom `Cross.toml` with pinned image) so a cross failure isn't mistaken for a backend-swap trigger.

## Decisions worth remembering

1. **For decision-only sprints, "Implementation notes for next sprint" should be explicitly framed as non-binding guidance, not a 7-step plan.** Otherwise the ADR pre-decides the next sprint's design and the next sprint becomes execution rather than design.
2. **The throwaway probe project is a cheap and honest validation pattern.** ~5 minutes of work converts a vague "I think this dependency works" claim into a sourced "I built it; here's the size and link table." Especially useful for ADRs where the load-bearing claims are about external systems.
3. **The `[features]` gate for `rusqlite` in `orchestrator/Cargo.toml` is now precedent.** Heavy dependencies that only some binaries need should be feature-gated by default. Saves binary size in the release tarball without affecting code clarity.
4. **The "operational vs. replay-verified" column distinction generalizes.** Any future schema in any future ADR for this codebase should annotate columns this way.
