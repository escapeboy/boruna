## What worked

1. **Single-shared-parser invariant nailed it.** The whole sprint's
   premise was "validate-vs-run drift is structurally impossible."
   The architecture made that load-bearing: `boruna run`,
   `boruna workflow run/resume`, `boruna policy validate`, and the
   MCP `boruna_run` object form all call
   `boruna_vm::policy_validate::parse_file` (or `parse`). Test
   `run_with_invalid_policy_propagates_kind` and
   `workflow_run_with_invalid_policy_propagates_kind` enforce it
   at runtime. Without that single-parser invariant the sprint
   would just have been "yet another validator."

2. **Two-pass validation (manual walk + typed deny-extra) was the
   right call.** `serde_json::Error` from `deny_unknown_fields` is
   awkward to map to stable `error_kind` strings — the message
   format is what's stable, not a structured enum. Manually
   walking the `serde_json::Value` first, with explicit allow-lists
   per object level, gives precise dotted paths
   (`net_policy.foo`, `rules.net.fetch.extra`) and a clean enum
   variant. The typed deserialization is then defense-in-depth.

3. **`accept_all_canonical_capabilities` is a future-proofing
   trip-wire.** When sprint 0.6.x adds a 12th capability, this
   test fails first. The fix is mechanical (add to fixture) but
   it forces a conscious update — exactly what convention #2
   ("strings stable forever") needs to anchor itself against.

4. **Adversarial review (`ce-correctness-reviewer`) caught two HIGH
   findings before merge.** Both were validate-vs-run drift in
   `boruna workflow run/resume --policy <file>` — the lenient
   `serde_json::from_str(&pjson)?` slipped through my own grep
   pass during build. The reviewer's prompt asked specifically
   for drift sites; that focused query found them in seconds.
   **Pattern: when a sprint has a single load-bearing invariant,
   ask the adversarial reviewer to verify it explicitly.**

5. **Latent `Policy::default()` bug surfaced via the round-trip
   test.** The derived `Default` produced `schema_version: 0`
   (u32 default) while serde's `#[serde(default = "...")]`
   produced 1. Invisible until the strict validator rejected 0,
   which made the existing `parse_policy_round_trip` MCP test
   fail. Fixed with a manual `Default` impl that matches serde's
   default. **Lesson: round-trip tests are how you find divergence
   between `Default` and `Deserialize`-with-defaults.**

## What didn't work / surprises

1. **The first workflow regression test failed because the workflow
   loader fails before the policy parse** in `boruna workflow run
   /no/such/wf --policy <bad>`. The CLI reads `workflow.json`
   first, so a non-existent dir errors with `cannot read
   /no/such/wf/workflow.json` before reaching the policy code.
   Fix: spin up a minimal valid workflow in a tempdir for the
   test. **Reordering the policy parse to happen FIRST would also
   work and is arguably better UX (fail-fast on the cheaper
   parse), but that's scope creep — deferred.**

2. **Pre-existing parallel-test flakiness in the orchestrator
   metrics test** (`export_returns_well_formed_prometheus_text_for_realistic_run_set`)
   surfaces from sprint 0.4-S14's `unsafe { std::env::set_var("BORUNA_ENV", ...) }`
   in `orchestrator/src/metrics.rs`. Not caused by this sprint;
   passes consistently in isolation. Per project conventions
   anti-pattern, this test should be removed or refactored.
   Logged in the rolling memory's debt list.

3. **Documentation footprint surprised me.** The reference doc
   (`docs/reference/policy-schema.md`) already existed from a
   prior sprint — I had to update it (not write it from scratch),
   including reversing prior copy that said "aliases are
   accepted." Convention #18 ("document in 4 places") matters
   exactly when you're tightening an existing surface, not just
   when adding a new one.

## Decisions worth remembering

1. **Schema-version is locked at first ship; only `1` accepted
   forever (until v2 = breaking).** New optional fields can be
   added at v1; shape changes require a version bump. The
   validator rejects 0 (the old `Default`) AND any value > 1.
   This forces operators to think about the contract before
   adding something custom.

2. **The legacy `invalid_policy` MCP error_kind stays alive** for
   non-object inputs (string typos, arrays, numbers). The new
   `policy.*` kinds are additive over it, not a replacement.
   Switching from `invalid_policy` to `policy.parse_error` for
   the case `{ "default_allow": "yes" }` was a deliberate refinement
   per convention #2 (stable strings, additive new ones); existing
   tests that expected `invalid_policy` for that case were updated
   to expect the more specific kind.

3. **CLI exit codes:** `0` ok, `1` IO error, `2` validation error.
   Not codified across the rest of the CLI (most subcommands
   collapse to ok / non-zero), but for `policy validate` as a CI
   gate the distinction matters: a missing file is an operator
   problem, a malformed file is a code-review problem.

4. **Deferred: `boruna policy lint`.** Warnings (e.g. `default_allow:
   true` + empty rules) are explicitly NOT errors in this sprint.
   Lint vs validate is a future split; documented in the design
   doc's open-questions section so a future sprint inherits the
   reasoning.
