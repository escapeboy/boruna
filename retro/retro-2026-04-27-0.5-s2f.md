## What worked

1. **Path 1 (client-side wait) was the right architectural
   choice.** The decision was already locked in the
   `distributed-execution-architecture` memory and the
   0.5-S2e retro recommendation. Following it preserved the
   "coordinator is dumb transport" invariant — the
   coordinator binary that ships 0.5-S2f is byte-identical
   to the one that shipped 0.5-S2e for the workflow-DAG
   logic. All advancement is in a new, isolated
   `advance_run_one_tick` function.

2. **Race-safe persistence primitive instead of
   overloading `upsert_step_checkpoint`.** The existing
   upsert hard-overwrites status on conflict. Reusing it
   would have voided in-flight worker leases when the wait
   client raced with claim/dispatch. Adding
   `insert_pending_step_if_absent` (using
   `INSERT ... ON CONFLICT DO NOTHING`) is the right
   primitive — and it's one regression test
   (`insert_pending_step_if_absent_preserves_running_row`)
   away from a load-bearing invariant.

   This is project-conventions §14 in action: the right
   conflict-resolution semantics for the right code path.
   Don't reach for the existing primitive when the new
   primitive's semantics are different.

3. **`prepare_persistent_run` factor (sprint 0.5-S2e)
   paid back AGAIN.** Adding the `workflow_def` field
   to metadata required a single helper
   (`embed_workflow_def_for_metadata`) called from both
   metadata-construction sites. The submit-only/
   non-submit-only branching is one if-else inside the
   helper. No call-site refactor.

4. **Adversarial review caught CORR-5
   (overloaded exit code 2).** Operators who script
   `coordinator wait` against a CI deadline need to
   distinguish "wrong run id" from "timeout". Splitting
   into 2 (error) and 3 (timeout) is one minute of work
   that prevents an operator-triage footgun from
   shipping. Convention §29 confirmed.

5. **The other 3 HIGH findings were not-a-bugs after
   careful trace.** Reviewing them was not wasted —
   walking through CORR-2 (retry semantics) confirmed
   that distributed retry is genuinely deferred to
   0.5-S3+ and the design doc already says so. Walking
   through CORR-4 (wave-1 approval-gate hangs) confirmed
   that submit-only's existing rejection covers it.
   These reviews caught misunderstandings I might have
   later, even if they didn't surface bugs.

6. **Dispatching plan + architecture + test-plan docs
   in parallel via three Write calls in one message.**
   Saved ~30 seconds of round-tripping. Sprint pipelines
   can absorb a lot of this kind of micro-parallelism.

## Patterns reinforced

- **Race-safe vs. clobber-safe persistence primitives.**
  When a code path may race with another writer that
  legitimately updates a row's status, prefer
  `INSERT ... ON CONFLICT DO NOTHING` over an upsert that
  would overwrite. The wait client's needs are different
  from the submit-only client's needs even though both
  write Pending checkpoints.

- **Per-item size caps on every variable-size embedded
  payload.** `step_sources` got 256 KiB per-item + 4 MiB
  aggregate caps in 0.5-S2e. `workflow_def` got 1 MiB in
  0.5-S2f. The pattern is: explicit ceiling, typed
  Validation error at submit time, regression test that
  exceeds the cap.

- **Documenting limitations in 4 places (per project-
  conventions §18).** Distributed-mode-retry-deferral is
  in: design-coord-wait.md (Documented limitations),
  CHANGELOG.md (Known limitations), the field doc on
  `PersistedRunMetadata::workflow_def`, and now this
  retro. When 0.5-S3 wires retry through, all 4 places
  get updated in lock-step.

- **Triaging adversarial-review findings honestly
  beats accepting all of them.** CORR-1 through CORR-4
  were HIGH severity but not bugs after careful trace.
  Mechanically applying every reviewer suggestion would
  have ballooned the sprint scope and added defensive
  code that doesn't pay back. Convention §29 says
  adversarial review pays for itself; convention §28
  reminds us the reviewer's reach can exceed the
  reviewer's grasp.

## What's NEXT

The 0.5.0 cycle has substantively shipped. End-to-end
multi-wave workflows now execute on a coord+workers
cluster. Future work falls into 0.5-S3+ themes:

- **0.5-S3 — auth.** mTLS / shared secret / OAuth.
  Mandatory before any non-loopback bind.
- **0.5-S4 — `boruna workflow run --coordinator <url>`.**
  Combines submit + wait into one command. Useful for
  CI; less useful for production (operators usually
  want to fire-and-forget then poll).
- **0.5-S5 — distributed retry policies.** Wires
  `RetryPolicy` through the coord/worker protocol so a
  failed step can be re-queued (instead of permanently
  Failed in distributed mode).
- **0.5-S6 — distributed approval-gate / external-
  trigger.** Today these step kinds are rejected at
  submit time; making them work in distributed mode
  needs the operator-bridge protocol from 0.3-S15
  generalized.
- **0.5-S7 — output blob references.** Today step
  outputs travel inline through metadata + HTTP
  responses. Large outputs (LLM responses) need a
  content-addressed blob store + references.

Each is a clear API boundary. Each can ship in one
session. The split-by-API-boundary pattern from 0.5-S2
should keep applying.
