## What worked

1. **Splitting 0.5-S2 into a/b/c was the right call.** Shipping
   the persistence state machine first (0.5-S2a), then the
   HTTP layer (this sprint), then the runner integration
   (0.5-S2c) means each sprint has a focused contract surface
   and a focused test target. The HTTP layer is genuinely just
   a thin wrapper — most of the correctness work happened in
   0.5-S2a.

2. **Reusing the 0.4-S15 strict-policy-validator at the worker
   boundary closed the validate-vs-execute drift surface.**
   Workers parse `policy_json` via `boruna_vm::policy_validate::parse`
   — the same function the CLI uses. Any policy that fails
   the CLI also fails the worker, with the same stable
   `policy.*` error_kind strings. **Pattern reinforced:** when
   the same data flows through validate and execute paths,
   both must call the same parser. The convention from 0.4-S15
   continues to pay back.

3. **Adversarial review caught 2 HIGH bugs that would have
   shipped silently.** F1 (worker swallowing 413) was a real
   correctness gap that turned into a production bug at
   0.5-S2c integration time — without the propagation fix, an
   oversize-output step would lease-expire and re-dispatch
   indefinitely, never failing. F2 (busy-spin on Ok(None))
   was a real DoS vector — under a misconfigured proxy
   returning 204 instantly, the worker would hammer the
   coordinator at ~10000 req/s.

4. **F5's deterministic-test pattern is reusable.** Testing
   distributed-state race conditions across processes is
   fundamentally hard. The cleanest pattern: *simulate* one
   half of the race via direct persistence-API calls, *exercise*
   the other half via the real CLI subcommand. The flagship
   test now uses `claim_step` to fake worker A's stale claim,
   then spawns worker B via the real HTTP path; the
   coordinator's startup sweep does the lease-expiry. **Add
   to convention list:** for distributed-state tests, prefer
   semi-real over fully-real when fully-real introduces races.

5. **`reqwest` with `default-features = false` + `rustls-tls`
   was the right dep choice.** Avoided pulling `openssl` and
   the system dep tree. The single new dep flowed cleanly
   into the existing `serve` feature without forking a new
   feature flag.

## What didn't work / surprises

1. **Adversarial review F1 was sneaky.** The original code
   "logged and moved on" for non-success responses. That looks
   defensive — operator gets a stderr line, presumably they'll
   investigate. But for the 413 case specifically, the
   worker holds a still-running lease until expiry, then the
   step gets re-dispatched, then the same 413 happens again,
   ad infinitum. The fix is non-obvious: report_fail with the
   error_kind in the message, so the step transitions to
   Failed and the run can progress (or terminal-fail per
   retry policy). **Lesson: silent-log-and-return-Ok is
   almost always wrong on a state-machine failure path.**

2. **The bulk-edit pattern from 0.5-S2a returned for the
   StepCheckpoint construction sites in the new test file.**
   I had to add the three new fields to the `populate_pending_step`
   helper and the F5 fix. The python-script approach from
   0.5-S2a is genuinely the right tool when adding a field
   to a struct that's constructed in 17+ places, but hand
   inserts work fine for the 2-3 sites in a single new file.

3. **The clippy `result_large_err` lint** fired on my
   `Result<_, Response>` helpers. I added `#[allow]` with a
   comment explaining the slow-path tradeoff. Small annoyance.

4. **The first integration test rev had a fundamental race.**
   The `worker_kill_mid_step_lease_expires_then_reclaim` test
   originally spawned a real worker A subprocess. The test's
   premise — "kill A mid-step before A completes" — assumed
   the step takes longer than the kill+detect cycle. For
   `fn main() -> Int { 99 }` the step finishes in microseconds.
   Under fast CI, A would complete before the test could
   even read runs.db, and the test silently degraded to a
   no-op assertion. Adversarial review caught it; the F5 fix
   simulates A's stale claim deterministically without ever
   running A as a subprocess.

## Decisions worth remembering

1. **Worker `report_complete` propagates non-lease_expired
   failures via `report_fail`.** Lease_expired is the only
   "silent retry" case (coordinator already re-dispatched).
   Everything else — 413, 5xx, network errors, unknown
   kinds — turns into a step failure so the run can
   progress. Documented in `report_complete` comment.

2. **100ms client-side backoff on empty claim responses.**
   Even with server-side long-poll, the client must have a
   floor. Documented as a constant with rationale.

3. **1s minimum for `extend_by_ms` at the HTTP layer.**
   `BAD_REQUEST` + `coord.invalid_request` for anything below.
   Closes the "lease deadline already past on response"
   footgun.

4. **Coordinator startup sweep is the only lease-expiry
   mechanism in this MVP.** No background timer yet —
   0.5-S2c will add one. Documented in CHANGELOG and design
   doc.

5. **Listener-merge with the dashboard is deferred to 0.5-S2c.**
   For now operators run the dashboard on a different port.
   The `bind_warning` field on `CoordinatorState` is reserved
   for the future merge.

## Patterns added this sprint

- **Worker non-success-response propagation.** Workers
  receiving any non-`lease_expired` failure on
  `/api/work/complete` must `report_fail` so the step doesn't
  strand. The "silent log and return Ok" pattern is almost
  always wrong on a state-machine failure path.
- **Client-side backoff on empty long-poll responses.**
  Even when the server cooperates with long-polling,
  misconfigured intermediaries can return 204 instantly.
  The client MUST have a floor.
- **Deterministic distributed-state regression tests via
  semi-real fixtures.** When testing race conditions across
  processes, simulate one half via direct persistence-API
  calls and exercise the other half via the real CLI
  subcommand. F5's pattern is reusable for any
  "kill-X-mid-Y" scenario.
- **`#[allow(clippy::result_large_err)]` is acceptable when
  the Err branch is genuinely the slow path.** Document the
  tradeoff in a comment.
