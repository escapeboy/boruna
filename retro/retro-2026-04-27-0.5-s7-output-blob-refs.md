# Retro — 0.5-S7 Output Blob References (2026-04-27)

**Sprint:** 0.5-S7 — Content-addressed offloading of large step outputs
**Branch:** sprint/0.5-s7-output-blob-refs (merged + deleted)
**Commit:** `27bcb3b`
**Master state:** ahead of origin by 4 commits; not pushed.

## Goal

Move step outputs >64 KiB out of the inline `step_checkpoints.output_json`
column into a content-addressed local blob store keyed by SHA-256, so
distributed-mode LLM steps stop pegging the coord SQLite when outputs
grow into the MB range. Audit hashes had to remain unchanged so existing
evidence bundles continue verifying.

## Outcome

Shipped end-to-end:

- New `orchestrator/src/persistence/blob_store.rs` — sharded
  `<root>/<aa>/<hash>` layout with atomic temp-rename writes,
  hex-only path-traversal guard, idempotent re-write.
- New `step_checkpoints.output_blob_ref` column via schema v3 → v4
  ALTER TABLE migration. Forward-included in `SCHEMA_V1_SQL`.
- New `RunCheckpointStore::read_step_output` accessor that resolves
  inline-or-blob transparently.
- `complete_step_cas` now decides offload at the threshold; CAS body
  unchanged otherwise.
- New bearer-gated, run-scoped coord HTTP route
  `GET /api/runs/{run_id}/blobs/{hash}`.
- New stable error_kind taxonomy: `coord.blobs.bad_hash`,
  `coord.blobs.not_found`.
- 31 new orchestrator tests + 5 new coord handler tests.
- All workspace gates green.

Audit hash chain is unchanged by construction (the ref IS the existing
output_hash). Replay of pre-S7 and post-S7 runs produces byte-identical
chains. Evidence bundle now resolves through the new accessor (review
fix H1 — was nearly a silent compliance regression).

## What worked

### 1. Adversarial review caught two HIGH findings before ship

`ce-correctness-reviewer` and `ce-data-integrity-guardian` ran in
parallel on the diff. Surfaced:

- **H1 (data-integrity-guardian):** `create_bundle` in
  `workflow/runner.rs:4531` read `cp.output_json` directly. For
  blob-stored outputs, that's None. The bundle would have shipped
  WITHOUT the large LLM output AND `evidence verify` would have
  PASSED — silent compliance corruption. Fix: switched to
  `store.read_step_output(...)`.

- **H2 (data-integrity-guardian + correctness-reviewer agreed):**
  `upsert_step_checkpoint` used `COALESCE` for both `output_json`
  and `output_blob_ref` independently. A row with pre-existing
  inline output, upserted with a blob ref, would end up with BOTH
  columns populated → `read_step_output` errors as Inconsistent →
  run becomes un-resumable. Fix: input-validation rejects
  both-set; SQL CASE expressions enforce mutual exclusion on
  conflict.

- **M1 (correctness-reviewer):** concurrent blob writers on the
  same hash shared a fixed `<hash>.tmp` filename, racing on the
  inode. Fix: per-writer unique tmp suffix (`<hash>.tmp.<pid>.<seq>`)
  using a `static AtomicU64` counter. Also switched to
  `create_new(true)` to surface unexpected pre-existence.

This is the third sprint where adversarial review caught at least
one HIGH-severity bug that would have shipped. Per project convention
§29, this is now a load-bearing pattern, not an optional polish step.

**Lesson reinforced:** when the design has both a write path and a
read-and-bundle path, list every read site explicitly during the
plan phase. The bundle path was named in the architecture doc but the
build phase only swapped the resume read path; the bundle read site
slipped through.

### 2. Schema invariant enforcement at the application layer plus SQL

Mutual-exclusion between `output_json` and `output_blob_ref` cannot
be a SQLite `CHECK` constraint added via `ALTER TABLE` (would require
table rewrite). The pattern that ended up being right:

1. **Reject at parse:** input validation in `complete_step_cas` and
   `upsert_step_checkpoint` returns `Inconsistent` if both inputs
   are non-null.
2. **Enforce at write:** SQL `CASE` expressions in the upsert clear
   the OTHER column when one is provided, so the ROW invariant
   never falsifies in the success path.
3. **Validate at read:** `read_step_output` returns `Inconsistent`
   if both columns somehow ended up populated (defensive read
   barrier for pre-existing corrupt rows).

Three layers, each with a different "why" — caller-bug detection,
ON CONFLICT correctness, defensive read. Per project convention §1
("reject at parse"), §14 ("pick the right primitive for the race
semantics" — bare assignment vs COALESCE).

### 3. Per-writer unique tmp suffix is the small-data clean fix

Initial implementation used `final_path.with_extension("tmp")` —
shared filename across writers. Reviewer caught the race. Fix was 6
lines:

```rust
use std::sync::atomic::{AtomicU64, Ordering};
static SEQ: AtomicU64 = AtomicU64::new(0);
let seq = SEQ.fetch_add(1, Ordering::Relaxed);
let pid = std::process::id();
let tmp_path = final_path.with_extension(format!("tmp.{pid}.{seq}"));
```

No `tempfile` crate dep needed; no per-blob random; no allocation
in the hot path beyond the `format!` for the suffix string. PID
disambiguates across processes; SEQ across threads in one process.
**Pattern to remember:** when you need a unique-per-call file
suffix in a low-frequency path, a static AtomicU64 + pid is enough.

### 4. Hard-coded threshold avoided a config-surface footgun

Was tempted to expose `BLOB_THRESHOLD` as a `Policy` field. Resisted
per project convention §1 (reject at parse, don't silently override).
A misconfigured threshold would silently change every operator's
audit-bundle shape — exactly the silent-failure-mode the platform's
trust story wants to avoid.

The threshold is exported as `pub const` so tests can reference it.
Future sprint can introduce a knob deliberately if a customer pulls
for it (per project convention §26, customer-pulled sprints can pull
forward roadmap items).

## What didn't work / what I'd do differently

### 1. The build phase needed a "find all consumers" pass

Searching for `output_json` reads is what surfaced the bundle bug
DURING REVIEW. That should have happened during build. Adding a
checklist item to the build phase: for any new persistence column,
grep all readers of the OLD column and explicitly classify each one
as "switch to accessor" or "stays inline-only".

This would have caught H1 in build, not in review. The CLAUDE.md
"Code Changes (MANDATORY)" section already says this for
renames/removals; it should explicitly cover **adding a new column
that mediates an existing one**.

### 2. ~26 StepCheckpoint literal sites needed updating

Adding a non-Default field broke 26 sites across the workspace. Did
this with a Python script via Bash:

```python
re.match(r'^(\s*)claim_id:\s*[^,]+,\s*$', line)
# insert "{indent}output_blob_ref: None,\n" if next line is `}`
```

Worked on first run. Took 2 minutes. **Lesson:** when adding a
required field, prefer adding `#[serde(default)]` on the field AND
making the constructor of the struct explicit (e.g. a `new()` or
`builder` pattern). Then literals don't proliferate. The
`StepCheckpoint` struct is full-public-fields by convention from
0.3-S2a; introducing a builder would be a separate refactor sprint.

For now, the script-replace approach is fine. Track as a debt:
"StepCheckpoint constructor / builder, breaks no API but reduces
diff cost on every additive column."

### 3. Toolchain skew on `cargo clippy --all-targets`

Local clippy (1.91.1) emits new lints on TEST CODE that pre-date
this sprint. The project CI runs `cargo clippy --workspace -- -D
warnings` WITHOUT `--all-targets` (per CLAUDE.md gates). So my new
code passed the gate as run by CI, but `--all-targets` flagged
~5 pre-existing test issues across `boruna-pkg`, `boruna-bytecode`,
`boruna-vm`, `boruna-framework`.

Per release-pipeline memory: "Treat the runner as canonical truth;
periodically rustup update locally." Local is at 1.91.1; runner is
at 1.95.0 per memory — runner is NEWER. So if these lints fire
locally they almost certainly fire on the runner.

**Followup task:** sweep test code for the pre-existing clippy
lints in a separate cleanup sprint (Sprint B?). Out of scope for
S7. Master CI may already be red on these — should verify.

## New patterns to add to project-conventions

### §37 — When adding a column that mediates an existing one, list all readers

When a new persistence column (e.g. `output_blob_ref`) mediates an
existing column's semantics (e.g. `output_json` is now sometimes
None even for completed steps), every reader of the existing column
becomes a potential silent-failure site. The build phase should
explicitly grep for direct field reads and switch them to a new
accessor (`read_step_output`) that resolves either source.

**Demonstrated:** S7 review caught `create_bundle` reading
`cp.output_json` directly. Resume path was switched in build but
the bundle path slipped through. Fix was one-line.

**Pattern generalizes** to ANY persistence schema change where the
old column's meaning shifts based on a new column's value.

### §38 — Per-writer unique tmp suffix for non-tempfile-crate atomic writes

When a code path needs atomic-rename file writes but doesn't justify
pulling in the `tempfile` crate, the pattern is:

```rust
static SEQ: AtomicU64 = AtomicU64::new(0);
let seq = SEQ.fetch_add(1, Ordering::Relaxed);
let pid = std::process::id();
let tmp_path = final_path.with_extension(format!("tmp.{pid}.{seq}"));
```

Pid + atomic counter is enough for cross-process and intra-process
disambiguation. Combined with `OpenOptions::create_new(true)`, it
also surfaces unexpected pre-existence as `EEXIST`.

**Demonstrated:** `BlobStore::write` in 0.5-S7. Reviewer caught the
shared-tmp race; this pattern is the small-diff fix.

### §39 — Mutual-exclusion column invariants need 3-layer enforcement

For pairs of columns where exactly one of (A, B) must be populated:

1. **Reject at API entry** — input validation surfaces caller bugs.
2. **Enforce at SQL** — `CASE` expressions in ON CONFLICT clauses,
   not COALESCE-on-each-column-independently (the COALESCE pattern
   silently allows both-set).
3. **Validate at read** — accessor returns Inconsistent if both
   columns somehow end up populated; defensive against pre-existing
   corrupt rows or future caller bugs that bypass layers 1+2.

**Demonstrated:** `(output_json, output_blob_ref)` mutual-exclusion
in 0.5-S7. H2 from adversarial review.
