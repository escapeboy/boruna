## What worked

1. **Existing `serve` feature flag was already wired.** The crate
   had axum/tokio as optional deps and `mod serve` for the
   framework-app server. Adding `mod dashboard` next to it cost
   zero new deps and zero feature-flag plumbing. **Lesson: when
   a project has an opt-in feature for one capability, the next
   feature in the same domain should opt into the same flag
   unless there's a reason not to. New flag = new permutation in
   CI; old flag = free.**

2. **Slim `RunSummary` was the right call.** Adversarial review
   flagged that returning full `RunRow` in `/api/runs` would
   multiply disclosure surface — every list response would expose
   every run's `policy_json` and `metadata_json`. The fix
   introduced a separate type that omits both fields; the
   per-run detail endpoint still returns the full record. **New
   pattern documented in conventions: list-vs-detail asymmetry
   for sensitive fields.**

3. **Banner-on-every-page invariant only existed in docs until
   the test caught the gap.** The design + reference docs both
   promised the bind-warning banner appears on every HTML page.
   The implementation rendered it on `/` but not on `/runs/:id`
   (different render function, hardcoded `""` for the banner
   slot). Adversarial review caught this with no live test —
   simple enough that a 2-line `_warns_when_bound_non_loopback`
   test per page would have caught it earlier. **Lesson: when a
   spec says "appears on every page," write one test per page
   asserting that. Multi-page features must enforce
   cross-page consistency at the test layer, not the doc layer.**

4. **Hand-rolled HTTP/1.1 client kept the test deps minimal.**
   For the CLI integration tests (`cli_dashboard.rs`) I avoided
   pulling `reqwest` or `ureq` as dev-deps by writing a 30-line
   `TcpStream`-based GET helper. The tests run in ~1s for all 8
   integration tests — fast enough that the spawn-and-kill
   pattern doesn't become a CI tax.

5. **`format_ms` negative-input bug came from defensive thinking
   in the wrong direction.** I added a `if ms == 0` guard for
   "missing timestamp" but didn't think through what `i64` could
   actually contain. Adversarial review traced the math — a
   negative `i64` walks the year loop with `days < 0`, exits
   immediately, then casts a negative day-count to `u32`,
   producing `1970-01-4294967295T...`. The `i64` type was the
   tell — if I'd used `u64` for "valid timestamp," I would have
   pushed the negativity question up to the caller. **Lesson:
   the type of a parameter is a contract. `i64` for timestamps
   means "anything goes including past-epoch and bad data";
   handlers must defend.**

## What didn't work / surprises

1. **`RunRow`/`RunRecord`/etc. didn't have `Serialize` derives.**
   I assumed they did because the orchestrator uses serde
   heavily. They don't — the audit and evidence pipelines
   serialize the inner `policy_json` / `metadata_json` strings
   pre-encoded, so the outer structs never went through serde.
   Adding `Serialize` was straightforward but a reminder to
   verify the actual derives before architecting around them.
   Captured in CHANGELOG with explicit "additive, non-breaking"
   note.

2. **Connection-reset transient flake on CLI integration tests.**
   One test (`cli_dashboard_serve_responds_to_index`) flaked once
   with `ConnectionReset`. The `wait_for_server` helper checks
   TCP connectability but the server's actual HTTP handler
   isn't necessarily ready when `bind` returns. Re-running
   passed cleanly. Acceptable: 1/N flake rate is below the
   threshold where I'd add retry logic.

3. **`useless_format` clippy lint** caught a `format!()` with a
   raw string but no interpolation. I had it as `format!(r#"..."#)`
   from copy-paste. One-line fix but a reminder: always run
   clippy on a feature-gated build, not just the default.

4. **Pre-existing 0.4-S14 parallel-test flake remains** — the
   `unsafe { std::env::set_var("BORUNA_ENV", ...) }` in
   `orchestrator/src/metrics.rs` test still occasionally trips
   under `cargo test --workspace`. Documented in the rolling
   memory's debt list. NOT introduced by this sprint.

## Decisions worth remembering

1. **Loopback-default is the right security stance for an MVP.**
   `--bind 0.0.0.0` is allowed for the operator who knows what
   they're doing, but the default protects the operator who runs
   `boruna dashboard serve` once on a public-facing host without
   reading the docs. The stderr warning + HTML banner are loud
   enough that the no-auth posture is hard to miss.

2. **Slim list type is the safer default, even when the detail
   type is verbose.** Dashboard list endpoints fan out the
   exposure across all rows — the detail endpoint is one
   deliberate fetch. Document the asymmetry and pick the slim
   type for the list.

3. **No template engine, no JS dependency.** The dashboard
   renders ~3 pages of HTML via `format!`. Adding `askama` or
   similar would be ~50 lines of build-time wiring for almost no
   benefit. The HTML is a server-side render of the JSON — the
   JSON is the contract; the HTML is convenience.

4. **`Serialize` derives on `RunRow`/`RunRecord`/etc. are now
   part of a public surface.** Adding the derives means any
   future field rename or removal becomes a breaking JSON-shape
   change. Documented in conventions: additive-only going
   forward for these types.
