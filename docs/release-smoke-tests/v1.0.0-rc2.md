Performed: 2026-04-28 (sprint W9-C)

## Scope

End-to-end verification that the GitHub Releases artifact for `v1.0.0-rc2`
(macOS arm64) works as published — checksum integrity, binary launches,
example workflow runs to completion, evidence bundle verifies.

This is the operator-facing smoke test from
[`Projects/boruna/v1.0-GA-checklist.md`](../../../barsy/obsidian/Barsy/Projects/boruna/v1.0-GA-checklist.md)
item #6 ("smoke test each release artifact on its target platform"),
performed for the macOS arm64 target. The two Linux musl targets remain
operator-side smoke tests on actual Alpine and aarch64 Linux hosts.

## Steps performed

1. **Download** `SHA256SUMS` from
   <https://github.com/escapeboy/boruna/releases/download/v1.0.0-rc2/SHA256SUMS>.
2. **Download** `boruna-1.0.0-rc2-aarch64-apple-darwin.tar.gz` from the
   same release.
3. **Verify** the SHA-256 digest of the downloaded tarball matches the
   line in `SHA256SUMS`:
   - Expected: `6d58de1c70843c09d011b0943c4c6236546343966be4b0287d433c503e1f4b2c`
   - Computed: `6d58de1c70843c09d011b0943c4c6236546343966be4b0287d433c503e1f4b2c`
   - **Match: ✓**
4. **Extract** the tarball; layout matches the documented release format
   (`boruna`, `boruna-mcp`, `boruna-orch`, `boruna-pkg`, `CHANGELOG.md`,
   `LICENSE`, `README.md`).
5. **Run** `boruna capability list`. Output verified:
   - `boruna 1.0.0-rc2` ✓
   - `capability_set_hash: sha256:980d017dc54e30c39c329484b501fbe46914d9ad344bfcb4610b0280f4300a67` ✓
   - `protocol_version: 1` ✓
   - 11 capabilities listed (matches the `Capability::ALL` table in the
     bytecode crate). ✓
6. **Run** an example workflow:
   ```sh
   boruna workflow run examples/workflows/llm_code_review \
     --policy allow-all --record --data-dir ./data
   ```
   - All 3 steps (`fetch_diff`, `analyze`, `report`) reported `Completed`.
   - Total duration: 30 ms.
   - Bundle written to `evidence/<run_id>/`.
7. **Verify** the evidence bundle:
   ```sh
   boruna evidence verify evidence/<run_id>
   ```
   - Output: `evidence bundle is VALID`. ✓

### ⚠ Doc bug caught and fixed

The `boruna capability list` output reported the random capability as
`random` (wire name from `Capability::name()`). My earlier `/content-review`
pass had inadvertently introduced `random.next` into:
- `AGENTS.md`
- `docs/concepts/capabilities.md`
- `docs/concepts/determinism.md`

These were corrected in commit `73e3429`. The fix is purely textual — no
code or release-artifact behavior was wrong; the binary always used
`random` as the wire name.
